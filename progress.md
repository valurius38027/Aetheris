### ًں”´ Critical Issues

#### C-01. KZG Proof Generation & Verification Mismatch
- **File**: `aetheris-zkp/src/lib.rs`
- **Problem**: `prove_conservation` calls `setup_params(k)` which generates a **new random CRS** every invocation via `ChaCha20Rng::from_seed(seed)` where seed contains "AETHERIS_TRUSTED". `verify_conservation` also calls `setup_params` independently, generating **different parameters**. Prover and verifier use different CRS â†’ verification **always fails**.
- **Impact**: All ZK proofs generated are unverifiable. The entire shielded transaction system is broken.
- **Fix**: Cache `ParamsKZG` and keys globally; ensure prover and verifier share the same CRS.

#### C-02. Curve Type Confusion (BN254 / Pasta / Vesta)
- **File**: `aetheris-recursive/src/lib.rs:29-33`
- **Problem**: Imports both `halo2curves::bn256::Fr as Fp` and `halo2curves::grumpkin::G1Affine as VestaAffine`, then treats `Fp` (BN254 scalar field) as Pasta `Fp`. The `fq_to_fp` function at line 51 is a no-op (`*fq`). `NonNativeChip` aims to do non-native arithmetic (Fp-in-Fq) but the type system is fundamentally misconfigured.
- **Impact**: All recursive circuit logic (`PoseidonChip`, `EccChip`, `NonNativeChip`) operates on inconsistent field types. Proofs generated are cryptographically meaningless.
- **Fix**: Choose one curve family (BN254/Grumpkin or Pallas/Vesta) and use it consistently.

#### C-03. `probe.rs` References Non-Existent Type
- **File**: `aetheris-recursive/src/probe.rs`
- **Problem**: `let _ = commitment::SomethingThatDoesNotExist;` â€” this type does not exist in `halo2_proofs`.
- **Impact**: **Compilation failure**. The entire `aetheris-recursive` crate cannot compile.
- **Fix**: Remove `probe.rs` or replace with a valid probe function.

#### C-04. Hardcoded Bridge Encryption Key
- **File**: `aetheris-ffi/src/lib.rs:32`
- **Problem**: The AES-GCM bridge key is hardcoded as `AETHERIS_SECURE_BRIDGE_2026_KEY!` (bytes `[0x41, 0x45, ...]`). All FFI communication between frontend (C#) and Rust kernel uses this static key.
- **Impact**: Zero security â€” anyone who reads the binary can decrypt all IPC traffic.
- **Fix**: Implement proper key exchange (e.g., X25519 handshake on first connect).

#### C-05. Fixed AES-GCM Nonce
- **File**: `aetheris-zkp/src/lib.rs:511`, `aetheris-ffi/src/lib.rs`
- **Problem**: `encrypt_note` uses `Nonce::from_slice(b"AETHERIS_NOT")` â€” a fixed 12-byte nonce. AES-GCM with a fixed nonce + same key produces **identical ciphertext** for identical plaintext, and worse, if the same nonce is used twice with the same key, the authentication key is compromised.
- **Impact**: Transaction note encryption is broken. Confidentiality and integrity guarantees are void.
- **Fix**: Generate a random nonce per encryption and include it in the output.

#### C-06. Genesis Phrase Leaked via FFI
- **File**: `aetheris-ffi/src/lib.rs:1236-1240`
- **Problem**: `aetheris_get_genesis_phrase()` returns the test seed mnemonic to any caller.
- **Impact**: Anyone can call this function to get the genesis wallet's mnemonic and steal the initial 21M AET.
- **Fix**: Remove this function in production builds.

### ًںں، Major Issues

#### M-01. `aggregate_proofs` Is a Hash, Not a Real Recursive SNARK (Resolved âœ…)
- **File**: `aetheris-zkp/src/lib.rs:373-422`
- **Problem**: Simulates recursive proof aggregation with `blake3` hashing + "recursive_snark_v2_" prefix. There is **no actual recursive ZK verification** happening inside the circuit.
- **Impact**: A malicious prover can forge an aggregate proof containing invalid transactions, as long as the Blake3 hash matches.
- **Fix**: Upgraded to Merkle tree commitment over individually verified tx proofs (Phase 4). Each proof is verified via `verify_conservation` (real KZG), then committed via Merkle root. No longer `#[deprecated]`.

#### M-02. VDF Modulus Is Not a Valid RSA Modulus (Resolved âœ…)
- **File**: `aetheris-crypto/src/vdf.rs:13-18`
- **Problem**: The hardcoded 2048-bit hexadecimal string contains repeating patterns (`291a47291a47...`) and was clearly generated by repeating sample text, not by multiplying two safe primes.
- **Impact**: The VDF's security relies on the assumption that squaring modulo an unknown RSA modulus is sequential. If the modulus has known factors, this assumption is broken.
- **Fix**: Replaced with RSA-2048 from the RSA Factoring Challenge (Phase 4) â€” a well-known semiprime whose factors are unknown to the public.

#### M-03. Dual P2P Implementation Conflict
- **Files**: `aetheris-node/src/p2p.rs` vs `aetheris-node/src/main.rs`
- **Problem**: `p2p.rs` defines a minimal `AetherisBehaviour` (gossipsub + kademlia + identify) and `AetherisNetwork` struct. `main.rs` defines a completely different `AetherisBehaviour` (with mdns + autonat + relay + dcutr) and runs its own event loop. `lib.rs` only re-exports `p2p.rs`.
- **Impact**: The node binary (`main.rs`) and the FFI library (`lib.rs` + `p2p.rs`) have incompatible P2P stacks. Features like NAT traversal, relay, and DCUtR are only in `main.rs`.
- **Fix**: Unify into a single `AetherisBehaviour` and single event loop.

#### M-04. `NonNativeChip::add` Carry Constraint Not Enforced
- **File**: `aetheris-recursive/src/lib.rs:1232-1257`
- **Problem**: The carry gate defines `carry * (1 - carry) = 0`, but the `synthesize` path in `add()` at line 1441 assigns the carry value to an advice column without using the gate's `s_add` selector. The carry constraint from `create_gate("nonnative_add")` is **never triggered**.
- **Impact**: Carries are unconstrained â€” overflow behavior is undefined in-circuit.
- **Fix**: Ensure `s_add` selector is enabled when assigning carry cells, or constrain carries via a dedicated region.

#### M-05. `LedgerState::restore_from_db` Full Replay at Startup
- **File**: `aetheris-node/src/state.rs:36-73`
- **Problem**: On every startup, all blocks from height 0 to current are re-deserialized and re-processed to rebuild `nullifiers`, `commitments`, and `all_outputs`. This is O(n) in block count.
- **Impact**: With 100K+ blocks, startup time becomes prohibitive (minutes to hours).
- **Fix**: Persist nullifiers/commitments as incrementally updated accumulator state.

#### M-06. Mining Thread Opens Separate DB Instance
- **File**: `aetheris-ffi/src/lib.rs:1696`
- **Problem**: The background mining thread creates its own `LedgerState` (and thus its own `sled::Db` handle) while the main thread also holds one. sled uses file locking, but concurrent write access from two handles can cause `IO error: lock conflict`.
- **Impact**: Mining crashes or corrupts the database under load.
- **Fix**: Share the `LedgerState` via `Arc<RwLock<>>` between threads.

### ًں”µ Minor Issues

#### m-01. Redundant `use` Statements & Dead Code
- Multiple files import unused items (e.g., `aetheris-recursive/src/lib.rs` imports `CStr`, `CString`, `c_char` from std but FFI functions use `#[no_mangle]` in the same file).

#### m-02. `ConstrainSystem` Column Bloat
- `ValueConservationCircuit` allocates 64 advice columns for bit decomposition (one per bit). This is extremely column-inefficient; lookup tables or fewer columns with multi-row decomposition should be used.

#### m-03. Hardcoded Constants Repeated Across Crates
- `VDF_DIFFICULTY`, `TARGET_BLOCK_TIME`, `EXPECTED_GENESIS_HASH` etc. appear in both `aetheris-core` and `aetheris-ffi`. Should be centralized.

#### m-04. `bincode` + `serde_json` Mixed Serialization
- P2P messages use both `bincode` and `serde_json` inconsistently (e.g., `main.rs` sends `BlockProposal` as JSON, while `p2p.rs` uses `bincode` for blocks). This causes deserialization failures between different node types.

## Stage 24 â€” Class Group VDF compose/hash-to-form Bugfixes (2026-06-01)

**Scope**: Fix two critical bugs discovered during class group VDF implementation: `hash_to_form` seed collision and `compose` CRT formula error.

### Bugs Fixed

#### B-1. hash_to_form seed collision (VDF 2048-bit discriminant)
- **Root cause**: For discriminants where `|D| â‰، 11 (mod 12)`, k=3 always works for any odd b. The loop `for k in 2..=k_max` always found k=3 first, and the reduced form collapsed to `(3, 1, c)` regardless of seed.
- **Fix**: Changed k-loop to start from `k_min = k_seed % 200 + 4` (range 4-203), skipping the problematic k=3. Different seeds now find different k values â†’ genuinely different classes â†’ VDF wrong-seed test passes.

#### B-2. compose CRT formula: missing `/ d`

- **Fix**: Added `/ d` to x0 formula. Also updated Python prototype.

### Phase 1.5.6 â€” Deferred Issue Remediation (Post-Phase-1.5 cleanup)

4 of 6 deferred issues from Phase 1.5 review were addressed:

**S-1 (Issue 1)**: Mining thread refactor (`aetheris-ffi/src/lib.rs:1783-2008`).
The pre-fix code created a `LedgerState::new_with_db(db.clone())` local copy at the start of every loop iteration, then called `apply_block` on the local copy (not on the canonical `state.ledger`). This diverged from the in-memory state used by the swarm task and FFI callers, allowing concurrent `apply_block` calls at the same height to race on the DB key. The fix: snapshot `(last_hash, current_height, current_difficulty, state_root, last_aggregate_proof)` under STATE lock and drop the guard; do VDF, mempool drain, and IPA accumulator fold off-lock; re-acquire STATE lock and apply on `state.ledger.as_mut().unwrap()` with a race-guard `if ledger.height != current_height { continue; }` (re-iterate if chain advanced during VDF). The `restore_from_db()` call in the lost-bid case was removed (no longer needed because `state.ledger` is always in sync via the swarm task).

**S-2 (Issue 4)**: Genesis state_root fix (`aetheris-ffi/src/lib.rs:342`).

### Phase 1.7 â€” Discriminant Boundary Check in `VDF::verify`

Fixes the pre-existing `debug_assert_eq!` panic at `aetheris-crypto/src/classgroup.rs:112` ("compose: discriminant mismatch") and the related `debug_assert!` at `classgroup.rs:139-142` ("CRT exact division failed"). Both fired in debug builds when a deserialized form had a valid-looking but mismatched discriminant. The release build silently produced garbage and returned `false` from `left == y`, but in debug builds (CI, tests) the panic would crash the test.

**Root cause**: `VDF::verify` deserialized attacker-controlled forms via `Form::from_bytes` (which doesn't validate discriminant) and passed them directly to `compose` / `pow`. A malicious proof with a wrong-discriminant form (which is easy to construct since the class group discriminant is a public constant `b"Aetheris Class Group VDF v1"`) would trigger the assertions.

**Fix (Option B, surgical boundary check)**: `aetheris-crypto/src/vdf.rs:110-117`. The `Form::from_bytes` matchers now guard `if f.abs_discriminant() == self.discriminant`. On mismatch, `verify` cleanly returns `false` without ever calling `compose`. Both `result` and `proof` are validated.

**Why Option B (boundary check) over Option A (change `compose` signature)**:
- Surgical: 4-line change in 1 file
- Catches at the only attacker-controlled entry point (the proof bytes)
- Zero impact on `Form::compose`'s library contract
- Transitively protects both `classgroup.rs:112` and `classgroup.rs:139-142` assertions (both reachable only via the `compose` call chain from `VDF::verify`)
