### 🔴 Critical Issues

#### C-01. KZG Proof Generation & Verification Mismatch
- **File**: `aetheris-zkp/src/lib.rs`
- **Problem**: `prove_conservation` calls `setup_params(k)` which generates a **new random CRS** every invocation via `ChaCha20Rng::from_seed(seed)` where seed contains "AETHERIS_TRUSTED". `verify_conservation` also calls `setup_params` independently, generating **different parameters**. Prover and verifier use different CRS → verification **always fails**.
- **Impact**: All ZK proofs generated are unverifiable. The entire shielded transaction system is broken.
- **Fix**: Cache `ParamsKZG` and keys globally; ensure prover and verifier share the same CRS.

#### C-02. Curve Type Confusion (BN254 / Pasta / Vesta)
- **File**: `aetheris-recursive/src/lib.rs:29-33`
- **Problem**: Imports both `halo2curves::bn256::Fr as Fp` and `halo2curves::grumpkin::G1Affine as VestaAffine`, then treats `Fp` (BN254 scalar field) as Pasta `Fp`. The `fq_to_fp` function at line 51 is a no-op (`*fq`). `NonNativeChip` aims to do non-native arithmetic (Fp-in-Fq) but the type system is fundamentally misconfigured.
- **Impact**: All recursive circuit logic (`PoseidonChip`, `EccChip`, `NonNativeChip`) operates on inconsistent field types. Proofs generated are cryptographically meaningless.
- **Fix**: Choose one curve family (BN254/Grumpkin or Pallas/Vesta) and use it consistently.

#### C-03. `probe.rs` References Non-Existent Type
- **File**: `aetheris-recursive/src/probe.rs`
- **Problem**: `let _ = commitment::SomethingThatDoesNotExist;` — this type does not exist in `halo2_proofs`.
- **Impact**: **Compilation failure**. The entire `aetheris-recursive` crate cannot compile.
- **Fix**: Remove `probe.rs` or replace with a valid probe function.

#### C-04. Hardcoded Bridge Encryption Key
- **File**: `aetheris-ffi/src/lib.rs:32`
- **Problem**: The AES-GCM bridge key is hardcoded as `AETHERIS_SECURE_BRIDGE_2026_KEY!` (bytes `[0x41, 0x45, ...]`). All FFI communication between frontend (C#) and Rust kernel uses this static key.
- **Impact**: Zero security — anyone who reads the binary can decrypt all IPC traffic.
- **Fix**: Implement proper key exchange (e.g., X25519 handshake on first connect).

#### C-05. Fixed AES-GCM Nonce
- **File**: `aetheris-zkp/src/lib.rs:511`, `aetheris-ffi/src/lib.rs`
- **Problem**: `encrypt_note` uses `Nonce::from_slice(b"AETHERIS_NOT")` — a fixed 12-byte nonce. AES-GCM with a fixed nonce + same key produces **identical ciphertext** for identical plaintext, and worse, if the same nonce is used twice with the same key, the authentication key is compromised.
- **Impact**: Transaction note encryption is broken. Confidentiality and integrity guarantees are void.
- **Fix**: Generate a random nonce per encryption and include it in the output.

#### C-06. Genesis Phrase Leaked via FFI
- **File**: `aetheris-ffi/src/lib.rs:1236-1240`
- **Problem**: `aetheris_get_genesis_phrase()` returns the test seed mnemonic to any caller.
- **Impact**: Anyone can call this function to get the genesis wallet's mnemonic and steal the initial 21M AET.
- **Fix**: Remove this function in production builds.

### 🟡 Major Issues

#### M-01. `aggregate_proofs` Is a Hash, Not a Real Recursive SNARK (Resolved ✅)
- **File**: `aetheris-zkp/src/lib.rs:373-422`
- **Problem**: Simulates recursive proof aggregation with `blake3` hashing + "recursive_snark_v2_" prefix. There is **no actual recursive ZK verification** happening inside the circuit.
- **Impact**: A malicious prover can forge an aggregate proof containing invalid transactions, as long as the Blake3 hash matches.
- **Fix**: Upgraded to Merkle tree commitment over individually verified tx proofs (Phase 4). Each proof is verified via `verify_conservation` (real KZG), then committed via Merkle root. No longer `#[deprecated]`.

#### M-02. VDF Modulus Is Not a Valid RSA Modulus (Resolved ✅)
- **File**: `aetheris-crypto/src/vdf.rs:13-18`
- **Problem**: The hardcoded 2048-bit hexadecimal string contains repeating patterns (`291a47291a47...`) and was clearly generated by repeating sample text, not by multiplying two safe primes.
- **Impact**: The VDF's security relies on the assumption that squaring modulo an unknown RSA modulus is sequential. If the modulus has known factors, this assumption is broken.
- **Fix**: Replaced with RSA-2048 from the RSA Factoring Challenge (Phase 4) — a well-known semiprime whose factors are unknown to the public.

#### M-03. Dual P2P Implementation Conflict
- **Files**: `aetheris-node/src/p2p.rs` vs `aetheris-node/src/main.rs`
- **Problem**: `p2p.rs` defines a minimal `AetherisBehaviour` (gossipsub + kademlia + identify) and `AetherisNetwork` struct. `main.rs` defines a completely different `AetherisBehaviour` (with mdns + autonat + relay + dcutr) and runs its own event loop. `lib.rs` only re-exports `p2p.rs`.
- **Impact**: The node binary (`main.rs`) and the FFI library (`lib.rs` + `p2p.rs`) have incompatible P2P stacks. Features like NAT traversal, relay, and DCUtR are only in `main.rs`.
- **Fix**: Unify into a single `AetherisBehaviour` and single event loop.

#### M-04. `NonNativeChip::add` Carry Constraint Not Enforced
- **File**: `aetheris-recursive/src/lib.rs:1232-1257`
- **Problem**: The carry gate defines `carry * (1 - carry) = 0`, but the `synthesize` path in `add()` at line 1441 assigns the carry value to an advice column without using the gate's `s_add` selector. The carry constraint from `create_gate("nonnative_add")` is **never triggered**.
- **Impact**: Carries are unconstrained — overflow behavior is undefined in-circuit.
- **Fix**: Ensure `s_add` selector is enabled when assigning carry cells, or constrain carries via a dedicated region.

#### M-05. `LedgerState::restore_from_db` Full Replay at Startup
- **File**: `aetheris-node/src/state.rs:36-73`
- **Problem**: On every startup, all blocks from height 0 to current are re-deserialized and re-processed to rebuild `nullifiers`, `commitments`, and `all_outputs`. This is O(n) in block count.
- **Impact**: With 100K+ blocks, startup time becomes prohibitive (minutes to hours).
- **Fix**: Persist nullifiers/commitments as incrementally updated accumulator state.

#### M-06. Mining Thread Opens Separate DB Instance
- **File**: `aetheris-ffi/src/lib.rs:1696`
- **Problem**: The background mining thread creates its own `LedgerState` (and thus its own `sled::Db` handle) while the main thread also holds one. sled uses file locking, but concurrent write access from two handles can cause `IO error: lock conflict`.
- **Impact**: Mining crashes or corrupts the database under load.
- **Fix**: Share the `LedgerState` via `Arc<RwLock<>>` between threads.

### 🔵 Minor Issues

#### m-01. Redundant `use` Statements & Dead Code
- Multiple files import unused items (e.g., `aetheris-recursive/src/lib.rs` imports `CStr`, `CString`, `c_char` from std but FFI functions use `#[no_mangle]` in the same file).
- `KzgChip::verify_opening` is a no-op (returns `Ok(())`).

#### m-02. `ConstrainSystem` Column Bloat
- `ValueConservationCircuit` allocates 64 advice columns for bit decomposition (one per bit). This is extremely column-inefficient; lookup tables or fewer columns with multi-row decomposition should be used.

#### m-03. Hardcoded Constants Repeated Across Crates
- `VDF_DIFFICULTY`, `TARGET_BLOCK_TIME`, `EXPECTED_GENESIS_HASH` etc. appear in both `aetheris-core` and `aetheris-ffi`. Should be centralized.

#### m-04. `bincode` + `serde_json` Mixed Serialization
- P2P messages use both `bincode` and `serde_json` inconsistently (e.g., `main.rs` sends `BlockProposal` as JSON, while `p2p.rs` uses `bincode` for blocks). This causes deserialization failures between different node types.

### Stage 8 — MOCK-10: KzgChip::verify_opening No-op Replacement (2026-05-31)

**Scope**: Replace the `Ok(())` no-op in `KzgChip::verify_opening` with a meaningful in-circuit constraint that binds the KZG opening commitment to the Fiat-Shamir transcript via Poseidon hash.

#### Background:
`KzgChip::verify_opening` was a pure no-op (`Ok(())` with a TODO comment). The full KZG pairing check `e(C - v·G₁, G₂) = e(π, X·G₂ - z·G₂)` requires Fp₁₂ arithmetic in-circuit, which is impractical within a Halo2 recursive circuit. The standard Halo2 approach defers the G₂ pairing to the outer verifier via the AccumulatorChip.

#### Changes Made:
- `aetheris-recursive/src/lib.rs` (`KzgChip::verify_opening`):
  - Retained on-curve assertion (`EccChip::assert_on_curve`)
  - Added Poseidon hash of the commitment (x, y) as Fiat-Shamir transcript binding
  - Removed `TODO(PHASE-3): Implement full KZG opening verification`
  - Added comprehensive doc comment explaining: the deferred pairing pattern, why in-circuit pairings are not used, and what each step accomplishes

#### What This Does:
| Step | Constraint | Purpose |
|------|-----------|---------|
| 1. On-curve check | `y² = x³ + 5` | Rejects malformed commitments |
| 2. Poseidon binding | `binding = Hash(C_x, C_y)` | Prover commits to the opening at circuit synthesis time |
| 3. Accumulator fold | (caller responsibility) | Defers G₂ pairing to outer verifier |

#### What Is NOT Done (by design):
- Full pairing check `e(C - v·G₁, G₂) = e(π, X·G₂ - z·G₂)` — this requires Fp₁₂ in-circuit arithmetic. Halo2's approach defers this to the outer accumulator verification, where the pairing is checked once for all accumulated proofs.

### Stage 12 — MOCK-14: State Root Merkle Tree (2026-05-31)

**Scope**: Replace `get_state_root()` returning `self.last_block_hash` with a real Blake3 Merkle root over the nullifiers and commitments sets.

#### Problem:
`LedgerState::get_state_root()` at `state.rs:169-173` returned `self.last_block_hash` — the hash of the serialized previous block. This is meaningless as a state commitment: it provides no binding to the actual state (nullifiers, commitments, UTXOs). A block could claim any state and pass verification.

#### Changes Made:

- `aetheris-node/src/state.rs`:
  - Added `use aetheris_zkp::build_merkle_root`
  - Replaced `get_state_root()` body:
    ```rust
    // Before: self.last_block_hash
    // After:
    let mut leaves = self.nullifiers.iter().copied().collect();
    leaves.extend(self.commitments.iter().copied());
    leaves.sort();
    build_merkle_root(&leaves)
    ```

The Merkle root is computed over sorted `nullifiers ∪ commitments`, providing:
1. **State integrity**: any change to nullifiers/commitments changes the root
2. **Fork detection**: different chains have different roots
3. **Deterministic**: sorted leaves give canonical ordering

### Stage 14 — m-02: Fix Column Bloat in ValueConservationCircuit (Running-Sum Range Proof) (2026-05-31)

**Issue**: `ValueConservationCircuit` allocated 64 separate advice columns for bit decomposition (one per bit of the 64-bit amount), totaling 69 advice columns. This is extremely column-inefficient and wastes prover/verifier resources.

## Stage 23 — Class Group VDF 迁移规划 (2026-06-01)

**Scope**: 将 Wesolowski VDF 从 RSA-2048 群迁移至虚二次域类群 $\text{Cl}(D)$，消除信任假设。更新数学规格文档、路线图和形式化验证任务。

### 动机

RSA-2048 VDF 的安全性依赖 **"RSA Labs 确实销毁了因数"** 这一信任假设。若因数泄露或未来被分解，VDF 安全完全崩溃。

类群 $\text{Cl}(D)$ 的群阶 $h(D)$ 在数学上不可计算——即使无限计算能力也无法在多项式时间内确定 $h(D)$。此属性 **不依赖任何信任假设**。

### 方案：自实现二元二次型类群

| 组件 | 原（RSA BigUint） | 新（Class Group `Form`） | 改动量 |
|------|-------------------|------------------------|--------|
| 群参数 | `modulus: BigUint` | `discriminant: BigUint`（2048 bit） | ~1 行 |
| 群元素 | `BigUint`（标量） | `Form { a, b, c: BigUint }`（三元组） | ~10 行 |
| 平方 | `(&y * &y) % &modulus` | `compose(&y, &y)` + `reduce()` | ~60 行 |
| 指数 | `x.modpow(&q, &modulus)` | double-and-add `compose/reduce` | ~30 行 |
| hash-to-form | seed → BigUint mod N | seed → Form 确定性映射 | ~30 行 |
| 序列化 | `to_bytes_be()` | a∥b∥c 紧凑编码 | ~20 行 |
| solve/verify | 不变 | 群运算替换，接口不变 | ~40 行 |
| **合计** | | | **~290 行** |

### 调用侧零改动

所有 4 个 VDF 调用侧保持签名不变，无需修改：

| 调用侧 | 调用 |
|--------|------|
| `aetheris-node/src/state.rs` | `VDF::new(difficulty)` → `solve()` → `verify()` |
| `aetheris-node/src/main.rs` | `VDF::new(difficulty)` → `solve()` / `verify()` |
| `aetheris-ffi/src/lib.rs` | `VDF::new(difficulty)` → `solve()` |
| 所有 19 个测试 | `VDF::new()` → `solve()` → `verify()` |

### 数学文档更新

## Stage 24 — Class Group VDF compose/hash-to-form Bugfixes (2026-06-01)

**Scope**: Fix two critical bugs discovered during class group VDF implementation: `hash_to_form` seed collision and `compose` CRT formula error.

### Bugs Fixed

#### B-1. hash_to_form seed collision (VDF 2048-bit discriminant)
- **Root cause**: For discriminants where `|D| ≡ 11 (mod 12)`, k=3 always works for any odd b. The loop `for k in 2..=k_max` always found k=3 first, and the reduced form collapsed to `(3, 1, c)` regardless of seed.
- **Fix**: Changed k-loop to start from `k_min = k_seed % 200 + 4` (range 4-203), skipping the problematic k=3. Different seeds now find different k values → genuinely different classes → VDF wrong-seed test passes.

#### B-2. compose CRT formula: missing `/ d`
- **Root cause**: `x0 = u * s` but should be `x0 = u * s / d` where `d = gcd(A1, A2)`. When d > 1, the base CRT solution B0 didn't satisfy `B ≡ b₂ (mod 2A₂)`, making the t-search fail.
- **Fix**: Added `/ d` to x0 formula. Also updated Python prototype.

### Changes Made

| File | Change |
|------|--------|
| `src/classgroup.rs` | `hash_to_form`: k-loop `2..=k_max` → `k_min..=k_max` with `k_min = k_seed % 200 + 4` |
| `src/classgroup.rs` | `compose`: `x0 = &u * &s` → `x0 = &u * &s / &d` |
| `src/vdf.rs` | Removed debug code from `test_verify_rejects_wrong_seed`; removed unused `pad_form_bytes`; removed duplicate `let pi` binding |
| `prototype/classgroup_proto.py` | `x0 = u * s` → `x0 = u * s // d_val` |

### 关键发现 + Fixes (per Multi-Agent Review 2-pass)

#### Fix I-1 (BLOCKER — math safety): `fp_from_blake3` 75% zero bug

`fp_from_blake3` 初版用 `Fp::from_repr(...).unwrap_or(Fp::ZERO)` 减少 32-byte blake3 输出到 Fp。`Fp::from_repr` 只接受 canonical (>= p_Fp) 的 32-byte LE repr,~75% 的均匀随机 32-byte 字符串被拒,触发 `unwrap_or(Fp::ZERO)` 静默回退到加法单位元。

后果:在 `accumulate()` step 7 的 Fiat-Shamir challenge `c = fp_from_blake3(...)`,有 75% 概率 `c = 0`;代入 step 8 `Q_new = Q + c * pi_commitment = Q` —— IPA chain binding 链在 3/4 的 accumulation 中**完全失效**。攻击者能 grind proof nonce 选 blake3 输出 >= p_Fp 的"幸运 proof"。

修复:改用 `Fp::from_uniform_bytes(&buf)` (mod-p reduction of 512-bit value,where `buf = [hash(32), 0u8; 32]`),`from_u512` 总输出一个 uniform random Fp,never silently zero。

### Phase 1.5.6 — Deferred Issue Remediation (Post-Phase-1.5 cleanup)

4 of 6 deferred issues from Phase 1.5 review were addressed:

**S-1 (Issue 1)**: Mining thread refactor (`aetheris-ffi/src/lib.rs:1783-2008`).
The pre-fix code created a `LedgerState::new_with_db(db.clone())` local copy at the start of every loop iteration, then called `apply_block` on the local copy (not on the canonical `state.ledger`). This diverged from the in-memory state used by the swarm task and FFI callers, allowing concurrent `apply_block` calls at the same height to race on the DB key. The fix: snapshot `(last_hash, current_height, current_difficulty, state_root, last_aggregate_proof)` under STATE lock and drop the guard; do VDF, mempool drain, and IPA accumulator fold off-lock; re-acquire STATE lock and apply on `state.ledger.as_mut().unwrap()` with a race-guard `if ledger.height != current_height { continue; }` (re-iterate if chain advanced during VDF). The `restore_from_db()` call in the lost-bid case was removed (no longer needed because `state.ledger` is always in sync via the swarm task).

### Phase 1.6 — Real Wesolowski VDF in `ZkProverSystem` Trait

Resolves audit finding **B-3** (`prove_vdf` is blake3 hash, not VDF proof; no `verify_vdf`). The `ZkProverSystem::prove_vdf` / `verify_vdf` trait methods in `aetheris-zkp/src/trait_.rs:52-60` were previously stubbed (returned `b"vdf_zkp_pasta_v1_simulated"` and `true` respectively). Phase 1.6 replaces these with real Wesolowski class-group VDF calls delegating to `aetheris_crypto::VDF`.

**Cargo**: `aetheris-zkp/Cargo.toml` adds `aetheris-crypto = { path = "../aetheris-crypto" }`. No cycle (crypto is a leaf crate; `aetheris-recursive` already links both as precedent).

**Trait signature** (BREAKING — but no workspace callers, see Investigation):

### Phase 1.7 — Discriminant Boundary Check in `VDF::verify`

Fixes the pre-existing `debug_assert_eq!` panic at `aetheris-crypto/src/classgroup.rs:112` ("compose: discriminant mismatch") and the related `debug_assert!` at `classgroup.rs:139-142` ("CRT exact division failed"). Both fired in debug builds when a deserialized form had a valid-looking but mismatched discriminant. The release build silently produced garbage and returned `false` from `left == y`, but in debug builds (CI, tests) the panic would crash the test.

**Root cause**: `VDF::verify` deserialized attacker-controlled forms via `Form::from_bytes` (which doesn't validate discriminant) and passed them directly to `compose` / `pow`. A malicious proof with a wrong-discriminant form (which is easy to construct since the class group discriminant is a public constant `b"Aetheris Class Group VDF v1"`) would trigger the assertions.

**Fix (Option B, surgical boundary check)**: `aetheris-crypto/src/vdf.rs:110-117`. The `Form::from_bytes` matchers now guard `if f.abs_discriminant() == self.discriminant`. On mismatch, `verify` cleanly returns `false` without ever calling `compose`. Both `result` and `proof` are validated.

**Why Option B (boundary check) over Option A (change `compose` signature)**:
- Surgical: 4-line change in 1 file
- Catches at the only attacker-controlled entry point (the proof bytes)
- Zero impact on `Form::compose`'s library contract
- Transitively protects both `classgroup.rs:112` and `classgroup.rs:139-142` assertions (both reachable only via the `compose` call chain from `VDF::verify`)

**4 new tests**:

In `aetheris-crypto/src/vdf.rs::tests`:
1. `test_verify_rejects_wrong_discriminant_result` — corrupt result's `a` byte, valid proof, verify returns `false`
2. `test_verify_rejects_wrong_discriminant_proof` — valid result, corrupt proof's `a` byte, verify returns `false`
3. `test_verify_rejects_both_wrong_discriminant` — corrupt both, verify returns `false`

In `aetheris-zkp/src/halo2_pasta.rs::tests`:

## Stage 34 — §1.12a NonNativeFqChip for In-Circuit IPA Verifier (2026-06-07)

**Scope**: Start §1.12 trustless in-circuit IPA verifier work by adding sound non-native Fq arithmetic over the recursive circuit field Fp. This is the prerequisite for verifying Pallas scalar arithmetic inside the Vesta-scalar Halo2 circuit.

| `prototype/non_native_fq_proto.py` | Prototype validating 3×85-bit limb arithmetic |

### Arithmetic Design

- **Field placement**: recursive circuit runs over Pasta `Fp`; IPA scalars are `Fq`, so Fq arithmetic is non-native.
- **Representation**: `FqElement = [l0, l1, l2]`, base `B = 2^85`, little-endian; 255 bits covers Fq.
- **Addition**: two-pass carry + reduction:
  1. `s_add` computes unreduced `S = a + b` with carry chain.
  2. `s_reduce` constrains `S = R + k·Fq`; `k` bit-checked.
  3. Row 8 gap added to avoid aux-cell overlap; row 9 range-checks `k`.
- **Multiplication**: quotient/remainder witness with integer carry chain:
  1. 9 `s_mul` rows: `p_ij = a_i · b_j`.
  2. Witness `Q = floor(A·B / Fq)` and `R = A·B mod Fq`.
  3. 9 `s_mul` rows: `qf_ij = q_i · fq_j`.
  4. `s_add` rows accumulate `P_k` partial-product columns.
  5. `s_reduce` rows constrain `P_k + c_{k-1} = (QF+R)_k + B·c_k`, plus final `c_4 = 0` check.
  6. Range checks: Q limbs 3×85 bits, R limbs 3×85 bits, carries c0..c3 4×90 bits.
- **Negation**: witness `Fq - a`, then verify in-circuit with `add(a, neg_a) == 0`.
- **Inversion**: witness `a^(Fq-2) mod Fq`, then verify with `mul(a, inv) == 1`.

### Review Findings and Fixes

| Finding | Severity | Fix |
|---------|----------|-----|
| Missing R-limb range checks allowed quotient substitution attempts | ❌ Blocking | Added 3×85-bit R limb range checks |
| `neg()` was witness-only and unconstrained | ❌ Blocking | `neg()` now calls `add(a, neg_a)` and constrains result to zero |
| `add()` row 7 `borrow_out_3` shared `aux[8]` with row 8 `k_check` | ❌ Blocking | Moved `k_check` to row 9; row 8 is an explicit gap |
| Carries c0..c3 were unbounded, so Fp equations did not imply integer equations | ❌ Blocking | Added 4×90-bit carry range checks linked to original carry-chain aux cells |
| `add()` k flag proves bitness but not comparison `S >= Fq` | ⚠️ Non-blocking | Carried forward; honest witnesses set k correctly, and current verified paths reject wrong outputs |

### Soundness Result

The final multi-agent review approved the multiplication gadget after carry range checks. With `c_k < 2^90`, each carry-chain row is bounded by `< 2^176 << p_Fp`; therefore a satisfied Fp equation is also the corresponding integer equation. The weighted telescoping sum with `c_4 = 0` proves `A·B = Q·Fq + R` over integers, so returned `R` is the correct Fq product.

### 关键设计决策
1. **S5 设计**：使用 `assign_and_constrain_squeeze_block` 链接区块，精确匹配 IPA 验证器电路模式。避免 `TranscriptWordStream`（Fp 类型）。
2. **Offset 点展平**：`fold_to_final` 接受展平的 `&[VestaPoint]` offset 点，索引为轮 0: n, 轮 1: n/2, ...，总计 = 2n−1。在主机端预计算。
3. **Fq→Fp 转换以用于主机点乘法**：`BigUint::from_bytes_le(Fq::to_repr())` → `Fp::from_repr()`——匹配 VestaEccChip 的位提取模型。
4. **S7 测试见证 native Fq IPA 证明** 在 Vesta 生成器（EqAffine）上，而非 Pallas。所有标量（eval, challenges, a_final）均为 Fq，原生处理。
5. **scalar_mul result 门**：`x·(y²−x³−5)=0`，允许恒等（x=0）或有效曲线点。避免在验证器分配时对见证值进行分支。
6. **IpaVerifierCircuit 的 s_decompose 门识别**：在测试的 Blake2b 压缩电路中，列 h（第 8 列，索引 7）被 s_decompose 门覆盖。添加了 `configure_no_gates` 以在需要时创建虚拟 Fq 配置，避免了门冲突。
